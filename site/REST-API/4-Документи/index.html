    Документи

    Документи — це об'єкти з бізнес-логікою: відвантаження, закупівлі, переміщення, оприбуткування тощо. Назва контролера вказана у полі ocontroller відповіді definition.

    Стани документа та поле flags

    Документ завжди знаходиться в одному з двох станів:

    [ Загальний ] ←── flags |= 128 ──[ Редагування ]
              ──── flags |= 4 ──→
    Стан Опис
    Загальний Документ зафіксований і доступний для читання всіма
    Редагування Документ відкритий для редагування і закріплений за поточним користувачем

    Бітова маска flags

    Біт Перевірка Опис
    1 (flags & 1) != 0 Документ помічено на видалення
    2 (flags & 2) != 0 Документ проведено — встановлюється системою автоматично
    4 (flags & 4) != 0 Документ в режимі редагування
    128 записується в save Сигнал зафіксувати документ

    Біт 2 (проведення) встановлює система. Клієнт не повинен встановлювати його вручну.

    Search

    POST /{controllerName}/search/

    Документи повертаються порціями ~50 штук, відсортовані за датою. Пагінація курсорна.

    Параметри mci, pa, own, gst для документів не застосовуються.

    Параметри запиту

    {
    "ft": "full",
    "f": 0,
    "df": 1,
    "rf": 0,
    "lastDate": "2026-04-10T12:00:00Z",
    "lastId": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
    "direction": -1
}
    Параметр Тип Опис
    ft string Рівень заповнення полів: "full", "small", "forlist", "none"
    f int Бітовий фільтр за flags — вказані біти повинні бути присутні
    df int Бітовий фільтр за flags — вказані біти повинні бути відсутні. Типово df: 1 щоб виключити видалені
    rf int Прапорці пошуку. rf: 32 — фільтрувати за видимістю відділів поточного користувача
    lastDate string | null ISO datetime останнього отриманого документа — курсор. Відсутній у першому запиті
    lastId GUID | null GUID останнього отриманого документа — курсор. Відсутній у першому запиті
    direction int | null Напрямок: -1 — старіші, 1 — новіші. Відсутній у першому запиті

    Приклади

    Перший запит — найновіші документи:

    POST /storageshipment/search/

{ "ft": "full", "df": 1 }

    Наступна порція — старіші від останнього отриманого:

    POST /storageshipment/search/

{
    "ft": "full",
    "df": 1,
    "lastDate": "2026-04-10T12:00:00Z",
    "lastId": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
    "direction": -1
}

    Новіші від точки (оновлення списку):

    POST /storageshipment/search/

{
    "ft": "full",
    "df": 1,
    "lastDate": "2026-04-10T12:00:00Z",
    "lastId": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
    "direction": 1
}

    Формат відповіді

    {
    "data": [
        {
            "id": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
            "document_datetime": "2026-04-10T12:00:00Z",
            "flags": 2
        }
    ],
    "message": null,
    "sw": 12
}

    Відповідь не містить загальної кількості документів. Ознака досягнення кінця — порція менша за ~50 елементів.

    List

    POST /{controllerName}/list/

    Повертає документи за переліком ідентифікаторів.

    Параметри запиту

    {
    "ft": "full",
    "ids": [
        { "id": "7c9e6679-7425-40de-944b-e07fc1f90ae7" }
    ],
    "rf": 0
}
    Параметр Тип Опис
    ft string Рівень заповнення полів
    ids array Масив об'єктів з полем id (internalGuid)
    rf int rf: 1 — повернути документ разом з табличними частинами

    Читання з табличними частинами

    За замовчуванням повертається лише шапка документа. Щоб отримати також рядки табличної частини — передати rf: 1:

    POST /storageshipment/list/

{
    "ft": "full",
    "ids": [{ "id": "7c9e6679-7425-40de-944b-e07fc1f90ae7" }],
    "rf": 1
}

    Save

    POST /{controllerName}/save/

    Для документів save — це покрокова модифікація: кожен запит змінює одне поле. Повна модель не передається.

    Поля-посилання на USAP-об'єкти передаються як GUID-рядок — не повний об'єкт.

    Переведення в режим редагування

    Встановити біт 4 у flags. Відбувається автоматично при будь-якій зміні поля. Явний виклик потрібен лише щоб захопити документ без одночасної зміни полів:

    POST /storageshipment/save/

{
    "data": {
        "id": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
        "flags": 4
    },
    "ft": "full"
}

    Зміна поля документа

    У data передається лише id документа та змінюване поле. Додатковий параметр ca (CurrentAction) містить назву поля — система визначає за ним перелік супутніх перерахунків:

    POST /storageshipment/save/

{
    "data": {
        "id": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
        "k": "aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa"
    },
    "ca": "k",
    "ft": "full"
}

    Приклади реакції системи залежно від ca:

    ca Що змінює користувач Що виконує система автоматично
    "cu" Валюта документа Виправляє контракт контрагента під нову валюту
    "p" Сума рядка Перераховує знижку
    "disc" Знижка Перераховує суму

    Конкретний набір реакцій залежить від типу документа і може відрізнятися між клієнтами.

    Фіксація документа

    Встановити біт 128 у flags:

    POST /storageshipment/save/

{
    "data": {
        "id": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
        "flags": 128
    },
    "ft": "full"
}

    Щоб зафіксувати і залишитися в режимі редагування — обидва біти одночасно (128 + 4 = 132):

    {
    "data": { "id": "...", "flags": 132 }
}

    Створення нового документа

    Передати нульовий GUID у id. Документ одразу відкривається в режимі редагування:

    POST /storageshipment/save/

{
    "data": {
        "id": "00000000-0000-0000-0000-000000000000"
    },
    "ft": "full"
}

    Сервер повертає GUID нового документа в data[0].id.

    Табличні частини

    Рядки документа зберігаються через окремий контролер з суфіксом lines. Кожен рядок має поле owner — GUID батьківського документа:

    POST /storageshipmentlines/save/

{
    "data": {
        "id": "00000000-0000-0000-0000-000000000000",
        "owner": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
        "nm": "guid-номенклатури",
        "q": 10,
        "p": 100
    },
    "ft": "full"
}

    Читання рядків — через /storageshipmentlines/list/ або /storageshipmentlines/search/.

    Повний цикл створення документа

    # 1. Створити новий документ
POST /storageshipment/save/   {"data":{"id":"00000000-0000-0000-0000-000000000000"},"ft":"full"}
# → отримуємо GUID: "aaaa-..."

# 2. Встановити контрагента
POST /storageshipment/save/   {"data":{"id":"aaaa-...","k":"guid-контрагента"},"ca":"k","ft":"full"}

# 3. Встановити склад
POST /storageshipment/save/   {"data":{"id":"aaaa-...","st":"guid-складу"},"ca":"st","ft":"full"}

# 4. Додати рядок
POST /storageshipmentlines/save/   {"data":{"id":"00000000-0000-0000-0000-000000000000","owner":"aaaa-...","nm":"guid-номенклатури","q":10,"p":100},"ft":"full"}

# 5. Зафіксувати
POST /storageshipment/save/   {"data":{"id":"aaaa-...","flags":128},"ft":"full"}

    Job

    POST /{controllerName}/job/

    Виконання іменованих команд над документами. Набір команд залежить від типу документа і може відрізнятися між клієнтами — кастомні команди створюються в рамках функціоналу розширень. Повний перелік не визначається через definition, необхідно уточнювати у постачальника рішення.

    Структура запиту

    {
    "ft": "none",
    "data": {
        "name": "назва_команди",
        "id": "7c9e6679-7425-40de-944b-e07fc1f90ae7"
    }
}
    Параметр Тип Опис
    ft string Зазвичай "none" якщо результат не потрібен
    data.name string Назва команди
    data.id GUID internalGuid документа

    Стандартні команди

    Команда Опис
    "copy" Скопіювати документ
    "delete" Позначити документ на видалення

    Приклад — копіювання

    POST /storageshipment/job/

{
    "ft": "none",
    "data": {
        "name": "copy",
        "id": "7c9e6679-7425-40de-944b-e07fc1f90ae7"
    }
}

    Відповідь містить GUID нового документа в extmessage.data[0]:

    {
    "data": [],
    "extmessage": {
        "data": [{ "id": "new-document-guid", ... }]
    },
    "message": null
}

    Формат відповіді

    • data — може бути порожнім або містити оновлені дані
    • extmessage — розширений результат (наприклад, GUID нового документа при копіюванні)
    • message — якщо не null — команда не виконалась, причина у значенні
